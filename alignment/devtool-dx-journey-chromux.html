Executive Findings

  • High confidence: Chromux now has a runnable and documented developer journey. The canonical report's claim that no runnable app exists is obsolete; version 0.30.2 includes source/package quickstarts, paired sessions, approval-gated previews, capture, persistence, recovery docs, proof artifacts, tests, and tags through chromux-v0.30.2.
  • High confidence: the configured development machine can complete the paired preview/capture/file-drop loop; a scripted real-app transcript exercises the actual PTY and live webview.
  • Low confidence: a fresh developer can reproduce that success. No clean Mac evidence covers prerequisites, native build, permissions, unsigned packaging, PATH/auth, delivery, and recovery.
  • High confidence: direct send is a separate one-off claude -p process, not injection into the paired live Claude, Codex, or Grok conversation.
  • Medium-high confidence: the product has credible daily-driver mechanics, but no sustained usage record proves productivity or retention.

The next 30–60 days should optimize proof: clean-machine setup, one supported verification path, explicit delivery continuity, and a five-day daily-driver record. Breadth should remain deferred.

Journey 1 — Source Install And First Launch

Route: Apple Silicon macOS → Node 22.12+ and Xcode CLT → cd prototype → npm install (Electron rebuild of node-pty) → npm start, or npm run install-app for an unsigned arm64 app that replaces /Applications/Chromux.app → create, detect, resume, or adopt an agent/shell session.

Strengths

  • Root and prototype quickstarts exist; the full guide names prerequisites and Claude delivery dependency.
  • Package metadata declares the Node engine and start, package, install, rebuild, smoke, capture, and targeted test scripts.
  • Finder launch intentionally uses login-shell behavior for PATH/auth; local source is recorded for managed updates.
  • DETECT, RESUME, FRESH, OPEN SHELL, saved projects, and restore reduce repeated setup.

Friction and risk

  • No clean-machine run proves prerequisites, native build, first launch, permissions, or delivery.
  • The root commands appear before prerequisites, unsigned/arm64 boundaries, and native-build remediation.
  • engines.node is documentation, not a purpose-built preflight.
  • The source-built unsigned arm64 installer is suitable for the named builder but not general public distribution.
  • Automation permission for terminal-tab titles is learned through a degraded DETECT result; process detection still works.

Next proof acceptance: a clean Apple Silicon Mac follows docs without intervention; missing/old Node, Xcode tools, native build, CLI, Automation, and storage failures each have an exact next action; elapsed time and Finder PATH/auth are recorded.

Journey 2 — Quickstart To First Success

Definition: start/adopt/resume a session; produce a loopback URL or local HTML path; see it queue without auto-navigation; approve it; inspect/capture; review bounded YAML and screenshot state; file-drop or send through one-off claude -p; confirm path and outcome.

Proof

  • The scripted real-app transcript passes actual PTY creation/output, file:// queue and OPEN, three console messages/one error, a second localhost preview queued without auto-open, YAML preview, screenshot, and file-drop.
  • The adapter is separately verified through the login shell; sample payload and screenshot are preserved.
  • The YAML v1 contract is bounded and manually retryable.
  • Preview-queue and capture-record suites pass in this review.

Remaining risk

  • The run is scripted on the configured machine.
  • Direct send creates a new one-off Claude process in the target cwd, not live-session injection.
  • Codex/Grok are first-class terminals, but direct send is Claude-specific; their reliable route is file-drop/manual handoff.
  • DETECT resume chooses the latest saved conversation by project; two agents in one cwd cannot be distinguished.

Near-term acceptance: one supported verification entry point emits a pass/fail summary; the modal states the one-off boundary before send; Codex/Grok show file-drop/manual handoff as the current supported path.

Journey 4 — Production Adoption For The Solo Builder

Production means real coding-agent use across several days without routinely returning to separate terminal/browser workflows—not enterprise deployment.

Present capabilities: local app packaging and managed updates; restore/retry; external detection and project-based resume; validated saved projects; favorites and paired browser state; attention, queue, activity, dynamic titles, and tabs; persistent themes; local-data inventory and delivery logs.

Unknowns/blockers: no sustained record of sessions, reviewed previews, useful captures, retries, wrong-session incidents, or exits from Chromux; the runnable-app condition in tasks/record-todo.md is satisfied but the record is unexecuted; no external report exists; unsigned arm64 source distribution limits broader OSS; captures/logs never expire and browser profile lacks in-app reset.

Gate: treat v0.30.2 as a daily-driver candidate until at least five real workdays are recorded and repeated friction is promoted.

Journey 5 — Team Rollout

Team rollout remains out of scope. There is no current support for shared workspaces, access controls, audit exports, managed retention, policy enforcement, signed deployment, administrator controls, or collaborative capture history. Revisit only after more than one developer repeatedly completes the solo route and requests shared evidence or managed policy.

Journey 6 — Retention And Repeated Use

Hypothesis: pairing, approval queues, attention state, capture artifacts, and recovery retain the user if they make parallel work easier to resume and supervise than separate windows.

Hooks: projects, favorites, restore, detection, resume/fresh, terminal links, queues, activity/attention, inspectable captures/logs, updates, and themes.

Missing evidence: no daily baseline, clean-machine time-to-success, useful-capture rate, retry rate, wrong-session rate, abandonment reason, or external first-success report. The repository supports a mechanism hypothesis, not a retention conclusion.

Prioritized 30–60 Day DX Backlog

  1. P0: archive a clean Apple Silicon install/first-success rehearsal covering prerequisites, timing, permissions, build, Finder launch, preview, capture, file-drop, delivery, and recovery.
  2. P0: put macOS/arm64, Node 22.12+, Xcode CLT, unsigned source build, and Claude-only direct delivery before root quickstart commands.
  3. P1: expose one supported first-success verification entry point around the existing smoke/E2E path.
  4. P1: state at the modal that direct send is separate one-off Claude; show file-drop/manual handoff for Codex/Grok.
  5. P1: execute the existing adoption record for at least five workdays and promote only repeated friction.
  6. P2: add first-run diagnostics for prerequisites, PATH, permissions, and storage only if rehearsal evidence shows repeated failure.
  7. P2: defer signed/universal distribution and automatic retention until controlled-preview evidence proves they block adoption.
