Executive Integration Stance

Chromux should integrate as a local macOS cockpit around existing terminal-native coding agents, not as a replacement runtime. Its v1 stack should preserve Codex and Claude Code CLI behavior, pair each active agent session with a browser preview pane, detect localhost and file:// previews, queue review events without stealing attention, and send explicit browser evidence back to the paired agent.

The most important integration decision is to keep the first loop narrow and inspectable: launch unchanged CLI agents, detect or explicitly open previews, inspect the paired browser pane, capture URL/selector/HTML/console/screenshot/session evidence, and deliver it through a transparent v1 adapter such as claude -p with file-drop fallback.

MCP, GBlockParty managed infrastructure, cross-machine sync, and team controls are future ecosystem surfaces. They should shape compatibility choices, but they should not become prerequisites for proving the daily-driver loop.

Required Integrations By Layer

Host app

macOS-first desktop shell, terminal panes, embedded Chromium preview, screenshot capture, file preview handling, and laptop-friendly focus discipline.

CLI agents

Codex and Claude Code launch as unchanged terminal CLIs with preserved shell behavior, auth state, project directories, and clear lifecycle errors.

Preview detector

Parse terminal output and explicit open events for localhost, loopback URLs, and local HTML paths; dedupe and route by originating session.

Review queue

Open or refresh when appropriate, but badge and queue background previews rather than hot-swapping the pane the user is reviewing.

Evidence payload

Capture schema version, timestamp, session ID, URL, title, selected selector/HTML, console tail, screenshot path, delivery target, and optional notes.

Delivery adapter

Start with transparent claude -p or file-drop delivery; defer live terminal injection and MCP until the basic loop works.

Storage, Privacy, And Local Data

Chromux's local-first trust story depends on documenting exactly what is captured, where it lives, and how it is deleted. The tool may handle local URLs, file paths, screenshots, selected DOM, console logs, auth-adjacent browser state, and project names. V1 should not upload captures by default, and "private by default" should not be claimed until storage, screenshot retention, profile state, and command invocation behavior are documented.

Future Ecosystem Surfaces

MCP is the natural later compatibility layer if Chromux needs to expose browser state to multiple agents or external clients. GBlockParty managed infrastructure is a deferred monetization and productization surface that becomes relevant only after Chromux becomes part of the builder's daily workflow.

Setup Path And First-Success Chain

  1. Install or run Chromux on macOS.
  2. Configure launch commands for Codex and Claude Code while preserving existing CLI auth.
  3. Start one or more terminal-native agent sessions in Chromux.
  4. Let a session emit a localhost URL or local HTML path.
  5. Detect the preview and route it to the paired browser pane or queue.
  6. Inspect the preview without leaving the cockpit.
  7. Select an element or page state and trigger capture.
  8. Review the generated YAML payload and screenshot path.
  9. Deliver the payload through the selected v1 adapter.
  10. Verify that the paired agent receives actionable evidence.

This path should become the README quickstart once runnable commands exist. Until then, it is an implementation acceptance path rather than user-facing install documentation.

Compatibility Constraints

  • macOS-first: the app, permissions, shell environment, and desktop UI should target macOS before cross-platform claims.
  • CLI auth state: Chromux should reuse existing CLI auth state and fail clearly when unavailable.
  • No custom agent runtime: the tool should orchestrate terminal agents rather than replace them.
  • Browser profile scope: cookies, local storage, and dev-server sessions need deliberate shared-vs-isolated profile decisions.
  • Sensitive local data: screenshots, DOM snippets, console logs, URLs, and paths may contain private project data.
  • Screen real estate: split panes must remain usable on a laptop, and review queues should avoid distracting hot-swaps.
  • Fork-base uncertainty: claims about cmux extensibility remain provisional until source inspection and a stack spike.
  • Payload size: console and DOM snippets must be bounded for prompt usability and privacy.
  • Delivery semantics: one-off delivery is not equivalent to live-session continuity.

Alternatives And Lower-Confidence Findings

Alternatives Considered

  • General-purpose browser in the terminal multiplexer: rejected because the wedge is session-paired review and evidence routing.
  • New agent runtime: rejected because existing CLI habits are a constraint and adoption advantage.
  • MCP-first architecture: deferred because it adds protocol complexity before the local capture loop is proven.
  • Hosted/GBlockParty-first architecture: deferred because monetization research keeps managed infra out of v1.
  • Immediate live-session stdin injection: deferred because it is riskier than proving payload generation and transparent delivery first.

Rejected Or Lower-Confidence Findings

  • Rejected: Browser automation primitives are unique to Chromux.
  • Rejected: MCP is required for v1.
  • Rejected: Chromux should own agent credentials.
  • Lower confidence: cmux can host the intended browser pane cleanly.
  • Lower confidence: claude -p will feel good enough.
  • Lower confidence: Future OSS users will accept the setup burden.
