Executive Findings

Docs Surface Improved

Chromux now has a root README, prototype README, payload contract, proof fixtures, and task queue. The prior canonical claim that no developer-facing docs exist is stale.

Main Blocker Shifted

The top blocker is no longer creating a README. It is splitting dense prototype README material into durable support and trust docs while preserving the first-success loop.

Troubleshooting Gap

A standalone prototype/docs/troubleshooting.md is still missing for preview detection, file previews, screenshots, console logs, CLI auth, wrong-session routing, and storage cleanup.

Local Data Gap

The docs make a useful local-data claim, but a complete prototype/docs/privacy-and-local-data.md is still needed before stronger public trust claims.

Proof Artifacts Present

The transcript, sample payload, and screenshot fixture exist. They should be linked from the prototype README as proof artifacts.

Migration Still Later

Migration docs from terminal plus browser, cmux, IDE previews, and browser MCP tools remain absent but are lower priority than support and trust docs.

Audit Findings

Q1 - First-success path is documented, but scanability is strained.

Evidence: root README gives cd prototype, npm install, and npm start; prototype README defines prerequisites, session start/adopt, preview routing, capture, YAML review, and delivery; prototype/package.json has matching scripts.

Impact: a developer can identify the path, but the prototype README carries onboarding, DETECT behavior, packaging, capture internals, troubleshooting, and storage in one long flow.

Canonical recommendation: mark quickstart docs materially improved, preserve the root quickstart, add a short "Verify the loop" checklist, and link proof artifacts near the first-success steps.

Q2 - Installable app path is documented, but support boundaries are mixed into README.

Evidence: prototype README documents npm run install-app, unsigned app behavior, /Applications/Chromux.app, and login-shell PATH behavior; package.json has the corresponding script.

Impact: sufficient for builder use, but unsigned app behavior, Automation permissions, CLI auth, and PATH belong in support docs before broader use.

E1 - Proof artifacts exist and cover the riskiest loop, but discoverability is one-way.

Evidence: transcript reports scripted E2E coverage of session creation, PTY output, file:// preview, console capture, queueing, capture modal, screenshot, file-drop payload, and separate claude -p adapter verification; sample YAML and screenshot exist.

Impact: proof basis is much stronger than the prior canonical audit, but README pages should directly surface these examples.

E2 - Missing failure-recovery example remains a proof gap.

Evidence: examples cover happy-path and queue behavior but not failed claude -p, missing auth, screenshot failure, wrong route, or manual retry.

Impact: support resilience is not yet proven as clearly as happy-path capture.

A1 - Capture payload schema v1 is documented with useful bounds and retention notes.

Evidence: payload docs define schema fields, nullable selection, console and screenshot sections, delivery metadata, bounds for DOM and console values, retention, and breaking-change versioning.

Impact: payload-contract blocker is resolved for v1. Add cross-links to the sample fixture.

A2 - Preview routing and delivery contracts are explained in prose but not reference-grade.

Evidence: prototype README explains loopback and absolute HTML path detection, empty-pane auto-fill, busy-pane queueing, popup queueing, throttled refresh, claude -p, and file-drop fallback.

Impact: good enough for first use; stable reference sections can come through troubleshooting or future preview-routing docs.

T1 - Troubleshooting exists in README form but should become standalone.

Evidence: README covers node-pty build failure, preview not detected, claude -p non-zero exit, screenshot missing, DETECT title permissions, and wrong resume target; tasks/todo.md still tracks standalone troubleshooting.

Impact: current bullets help, but self-service support needs symptom-oriented diagnostics, exact paths, retry steps, and escalation criteria.

T2 - Support-language evidence is still missing.

Evidence: approved scope says no external support evidence or user-language source is available; no issues, tickets, transcripts, or user quotes are in scope.

Impact: troubleshooting priorities are internally evidenced, not user-reported.

P1 - Local-data claim needs a complete data-handling reference.

Evidence: README documents capture paths, delivery log, browser profile partition, and says nothing leaves except what claude -p sends; task queue still calls for privacy/local-data docs.

Impact: add a data-handling reference naming captured data types, storage locations, retention/deletion behavior, and delivery boundary before stronger public claims.

M1 - Migration docs are absent and lower priority than support/trust docs.

Evidence: no migration docs exist; prior research identifies manual terminal plus Chrome, cmux, IDE preview panes, Playwright MCP, and Chrome DevTools MCP as baselines.

Impact: useful before public adoption, but not required for the builder's first loop.
