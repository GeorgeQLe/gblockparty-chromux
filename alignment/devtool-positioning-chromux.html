Executive Positioning

Chromux should be positioned as a personal-first macOS cockpit for developers supervising multiple terminal-native Codex and Claude Code sessions while reviewing generated localhost and local HTML output. The refreshed evidence makes the wedge narrower than "browser in the devtool" or even "agent can inspect a browser." Cursor, Codex, Claude Code, Playwright MCP, and Chrome DevTools MCP now cover substantial browser, agent, parallel-work, and debugging surfaces.

The durable positioning wedge is session-paired human review and evidence routing: each terminal agent session keeps its own paired preview context, new previews are queued without stealing attention, and the developer can capture the current browser state into an inspectable payload that is routed back to the correct session or delivery adapter.

Chromux is a local macOS cockpit for developers running parallel terminal coding agents: it keeps each session tied to its live preview and turns browser evidence into an explicit payload for the right agent.

Short variant: A local cockpit for parallel coding agents that keeps every terminal session tied to its preview and sends browser evidence back to the right agent.

Findings By Research Question

Q1 - Alternatives

The strongest alternatives are manual terminal plus Chrome, Codex CLI plus Codex app/browser surfaces, Claude Code CLI/Desktop/Web/Chrome, Cursor Agents Window plus Browser, Playwright MCP, Chrome DevTools MCP, IDE/browser preview panes, terminal multiplexers, and the intended but unverified cmux fork base.

The updated competitive picture is more crowded than the existing artifact implied. OpenAI Codex docs now list CLI, app browser, Chrome extension, Computer Use, Appshots, Worktrees, subagents, MCP, and cloud tasks. Cursor Browser covers logs, network, screenshots, inline/separate panes, dev-server awareness, design editing, persisted state, and enterprise controls. Claude Code spans terminal, IDE, desktop, browser, web, mobile, multiple agents, background agents, Chrome debugging, MCP, hooks, and claude -p. Playwright MCP and Chrome DevTools MCP expose browser automation/debug primitives directly to agents.

The gap remains that these alternatives are either IDE/product surfaces, automation servers, or single-vendor agent surfaces. None of the inspected sources position themselves as a small local cockpit whose primary object is many unchanged terminal sessions, each with a persistent paired preview and a human-controlled evidence payload routed back to the originating session.

Q2 - Unique Workflow Advantage

Browser control, screenshots, logs, network traffic, and browser panes are not unique. Chromux's defensible advantage is terminal-native session preservation, 1:1 session-to-preview pairing, preview discovery for localhost and file://, review queue behavior, explicit browser-to-agent capture payloads, and transparent delivery or file-drop fallback.

Q3 - Ecosystem Fit

Chromux should sit beside the agent ecosystem rather than compete as an IDE, cloud agent, automation protocol, or new runtime. Codex and Claude Code are appropriate first-class session types because both support terminal-native workflows. MCP is a compatibility path, not a v1 dependency, especially because Playwright MCP now distinguishes token-efficient CLI/skills workflows from persistent MCP browser loops.

Q4 - Trust Claims

Honest v1 claims are local-first desktop wrapper, unchanged CLIs, explicit user-triggered capture, inspectable local payloads, visible delivery logs, and no team/cloud/sync/telemetry implication unless implemented. Avoid "private by default," "secure browser automation," "works with any agent," or "replaces Cursor/Claude/Codex browser tools" until implementation and docs support those statements.

Q5 - Switching Cost

For the builder, switching cost is acceptable only if Chromux preserves CLI habits and beats manual terminal plus Chrome. For future OSS users, costs include macOS install, Electron/Chromium footprint, CLI auth reuse, local storage trust, browser permissions, preview detection, payload delivery semantics, and troubleshooting.

Q6 - Preserve Or Amend

Amend, but do not reverse, the existing statement. The old statement is directionally correct, but the safer positioning removes "unlike AI IDEs or generic browser automation" as the main contrast and replaces it with paired multi-session terminal cockpit plus explicit evidence routing.

V1 Priority Guidance

Differentiating

Session-preview pairing, preview discovery, non-attention-stealing queue, explicit capture payload, inspectable file-drop or claude -p delivery, wrong-session prevention.

Table-Stakes

Reliable terminal panes, browser navigation, screenshot capture, logs for capture/delivery, macOS setup, CLI auth and PATH diagnostics.

Defer

HAR/cookie capture, live terminal injection, MCP abstraction, visual editing, cloud/team/admin, monetization, broad "any agent" claims.

Source Coverage Gaps

  • No hands-on trials of Cursor Browser, Codex app/browser, Claude Code Desktop/Web/Chrome, Playwright MCP, or Chrome DevTools MCP were run.
  • No reliable primary cmux evidence was inspected in this checkout.
  • No runnable Chromux source, package manifest, payload schema, or demo transcript exists locally.
  • No external user interviews, installs, issues, support requests, or retention evidence exists.
  • Windsurf, Devin, and other AI IDE/cloud-agent products were not deeply re-reviewed because the approved scope prioritized terminal-agent, browser, MCP, and local artifacts.
  • Pricing and monetization comparison were intentionally out of scope for this positioning refresh.
