Executive Recommendation

Recommendation: Chromux should not monetize v1. The near-term stance should be free personal tooling, likely private or source-available until the daily-driver loop is proven, with an explicit option to become permissive OSS after the stack spike and capture loop work.

The first monetizable version should not default to "Chromux Pro." The stronger path is: keep the local desktop cockpit free; attach paid value only to hosted or managed surfaces when they exist; and use team and enterprise triggers as evidence gates rather than roadmap commitments.

This matches current market structure: AI coding tools commonly use free entry tiers, paid individual usage tiers, usage credits, and team/enterprise controls. Chromux does not yet have the expensive hosted compute or validated team workflow that would justify those mechanics.

Free / OSS Stance

Claim: V1 should be free and personal-first, with monetization explicitly deferred.

Evidence: Local docs define the builder as primary beneficiary and defer productization/GBlockParty monetization. Cursor, GitHub Copilot, PostHog, and Sentry all show free entry tiers for developer products.

Inference: Free entry is normal in devtool markets, and Chromux has an even stronger reason to be free because local-only v1 has low marginal infrastructure cost and no external validation.

Decision impact: Do not add licensing, plans, payments, account gates, cloud entitlements, or usage metering to v1.

Recommended stance: Start private or source-available while the builder validates the workflow. Move to permissive OSS once the architecture is stable enough that outside users can run it without support-heavy handholding. Avoid copyleft or artificial commercial restrictions unless a real hosted business emerges.

Usage Limits

Claim: V1 should use local resource guardrails, not commercial limits.

Sessions

No paid limit; warn when panes or agents exceed local machine comfort.

Capture history

Local retention setting with short default; no cloud quota.

Screenshots

Local path visibility and deletion controls; no upload by default.

Console logs

Bounded tail size per capture to avoid huge payloads.

Profiles/cookies

Isolate by session or project where practical; document persistence.

Future paid limits

Cloud sessions, managed browsers, retention, storage, seats, budgets, and GBlockParty infra minutes only after managed services exist.

Decision impact: Keep v1 limits framed as reliability and privacy controls. Do not condition local usage on a paid plan.

Team Conversion

Claim: Team conversion is plausible but unproven; the right trigger is repeated shared workflow demand, not "teams exist in competitor pricing."

  • More than one developer wants to inspect the same agent-generated preview or capture.
  • A team wants a shared review queue or persistent capture evidence across machines.
  • A manager/security owner asks for policy controls around agent browser access.
  • A company wants managed installation, update control, audit trails, or retention rules.
  • A GBlockParty customer wants a UI for managed agent/browser infrastructure.

Decision impact: Do not build team features until at least two triggers recur. If they recur, start with team workspace evidence sharing and admin controls, not collaboration theater.

Enterprise Triggers

Claim: Enterprise value attaches to control, observability, deployment, and risk management.

Identity

SSO/SAML/OIDC, SCIM, RBAC.

Audit

Browser captures, prompt payloads, screenshots, destinations, agent/session IDs.

Policy

Repository/project access, browser/network controls, model allowlists, BYO LLM.

Retention

Screenshot, DOM, log, cookie, capture, and artifact retention policies.

Deployment

Fleet deployment, managed auto-update, self-hosted cloud sessions or managed browsers.

Support

Budget caps, support SLAs, implementation help, security documentation.

Decision impact: Keep capture payloads auditable and storage boundaries clear, but do not implement enterprise surfaces in v1.

GBlockParty Managed-Infra Lens

Claim: GBlockParty is the best monetization lens only after Chromux becomes a daily-driver UI and there is a credible managed-infra offering behind it.

What GBlockParty could sell: managed browser/agent sandboxes, persistent remote sessions and previews, searchable capture history, team-scoped sandboxes, hosted runners with predictable concurrency and cleanup, and compliance/audit records around agent/browser actions.

Natural packaging: Chromux local app remains free. A GBlockParty account is optional and unlocks managed infrastructure. Pricing can be usage-based around sandbox hours, parallel workers, storage, and retention, with team seats for admin/audit features.

Risks: If local Chromux is not part of the builder's daily workflow, a GBlockParty UI is premature. If hosted infra is not materially better than local previews, account setup will feel like friction. If captures include sensitive screenshots, DOM, logs, cookies, or local URLs, privacy/retention/deletion controls are prerequisites.

Decision impact: Keep GBlockParty as a deferred lead-gen/managed-infra path. The revisit trigger remains: "Chromux is part of my daily workflow."

Unit Economics

Local-Only V1

Marginal cost is near zero per user if the app is local-only and ships no cloud services. Real costs are development time, support, signing/notarization, distribution, docs, issue triage, and compatibility testing. Apple Developer Program membership is a likely annual cost if distributing through Apple developer channels; Apple lists paid membership at 99 USD per membership year.

Paid Binary / Support

Possible costs include release signing/notarization, updates, support for Electron/Chromium, macOS permissions, CLI auth state, local ports, file permissions, documentation, troubleshooting, and regression testing. This can justify paid support later, but it is a poor first monetization surface.

Hosted / Managed Surface

Cost drivers include browser/agent sandbox compute, persistent storage for screenshots/captures/logs/traces/artifacts, network egress, model/API pass-through, team security/compliance work, support, and incident response. The model should be free local app plus optional managed infra credits, usage overages, team seats, and enterprise custom terms.

Rejected Or Lower-Confidence Findings

  • Rejected: Charge for local session count. This would monetize a local resource that costs Chromux nothing and would undermine the parallel-session value proposition.
  • Rejected: Team pricing should drive v1 architecture. Team willingness to pay is unvalidated, and current docs explicitly defer team/collab features.
  • Lower confidence: Permissive OSS is definitely best. It likely maximizes adoption, but the builder may prefer private iteration until the fork base and capture architecture settle.
  • Lower confidence: Paid binary/support can work. It can work only if installation demand exists and support load is manageable.
