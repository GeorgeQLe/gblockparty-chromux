3 · v1 scope vs deferred

In v1 daily driver

  • cmux fork, split-pane per agent session
  • Auto-open/refresh each session's localhost:PORT preview
  • Open local file:// HTML previews (no dev server)
  • Review queue — new previews badge/queue, never hot-swap the focused pane
  • One "Send to agent" button → claude -p with console logs + highlighted-element DOM/selector + screenshot + current URL

Deferred out of v1

  • Live-session stdin injection into the paired agent (fast follow after claude -p)
  • Full network/telemetry capture (perf entries, request waterfall)
  • Unified-sidebar layout toggle
  • Chromium/VSCode fallback bases (only if cmux blocks a first-class browser pane)
  • Productization / OSS release / GBlockParty monetization

4 · Riskiest unknowns carried forward

Pre-research — these are named, not solved. They become the first things the brief flags to validate before or during build.

  • unknown cmux extensibility: can cmux's Electron stack host a first-class embedded browser pane (BrowserView/webview) with the console/DOM/screenshot capture hooks? Validate against actual cmux source; Chromium or VSCode are the fallback bases if not.
  • unknown Payload → live session handoff: claude -p ships first (lowest-risk); reliably injecting a payload into a live PTY agent session (stdin) is the real open technical question, so it's deferred to a fast follow.
  • unknown localhost auto-detection: how Chromux reliably learns each session's emitted dev-server URL (log scraping, port watching, or a convention) to drive auto-open/refresh.
