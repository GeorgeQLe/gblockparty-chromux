Adaptive Questions

1. Which participants and Chromux surfaces are required in the first useful version?

Agent confidence: medium

Recommended default: require Codex and Claude Code as debate participants; treat Grok as a best-effort adapter; accept task/plan text, repository file references, proposed diffs, and bounded browser captures; render arguments, evidence links, verdict, dissent, and next action in a local Chromux review surface.

For the first useful version, require Codex and Claude Code participants, keep Grok as a best-effort adapter, accept task or plan text, repository file references, proposed diffs, and bounded browser captures, and render arguments, evidence links, verdict, dissent, and next action in a local Chromux review surface.

2. What orchestration and persisted-data boundary should the research optimize for?

Agent confidence: medium

Recommended default: Chromux owns a local debate state record and UI, invokes existing authenticated agent CLIs through explicit adapters, stores structured round metadata and references to evidence instead of duplicating whole repositories, and exports a human-readable Markdown verdict. Research should test whether Gigachadd can supply that state transition contract intact.

Optimize for Chromux owning a local structured debate state record and review UI, invoking already-authenticated agent CLIs through explicit adapters, storing round metadata and bounded evidence references rather than repository copies, and exporting a human-readable Markdown verdict. 3. What setup and migration experience is acceptable?

Agent confidence: medium

Recommended default: ship Debate / Challenge as an additive opt-in beta with no migration of current workspaces; discover already-installed agent CLIs and reuse their shell authentication; do not require users to install a separate Gigachadd binary unless research proves direct CLI integration materially preserves unique mechanics.

Use an additive opt-in beta with no migration of existing workspaces, discover installed agent CLIs and reuse their shell authentication, and avoid requiring a separate Gigachadd binary unless direct CLI integration is proven to preserve unique mechanics that a Chromux-native implementation would lose.

4. How should compatibility and provider asymmetry be handled?

Agent confidence: low

Recommended default: target Chromux's currently supported macOS/Electron environment and current installed CLI contracts; capability-detect each provider rather than promise identical flags or outputs; preserve the workflow when one optional participant is unavailable; record exact tested versions in the research rather than declaring broad compatibility.

Target Chromux's currently supported macOS and Electron environment plus current installed agent CLI contracts, capability-detect providers rather than assuming identical flags or output, allow the workflow to continue when an optional participant is unavailable, and record exact tested versions instead of claiming broad compatibility.
