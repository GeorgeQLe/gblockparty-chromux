Chromux Mobile — Prototype Gallery

Seven interaction directions for a phone-native Chromux: a remote agent command center for supervision, correction, approval, browser QA, and emergency intervention — not a miniature desktop IDE. Every prototype renders the same fabricated fleet so the directions are directly comparable. The desktop explorations live in the design gallery.

THE LOOP · see which agents need attention → inspect their latest work → intervene through SSH → operate the paired browser → capture evidence → return it to the correct agent
PROTO A
01 · A · DARK
Mission Control
Compact flight deck — graphite, warm amber, pale cyan instrument rules. Attention-first fleet dashboard, persistent Agent · Terminal · Browser · Evidence switcher, swipeable attention cards.
PROTO B
02 · B · LIGHT
Agent Inbox
Warm editorial paper — sessions as an inbox, conversational summaries with expandable activity blocks, a reply composer that previews exactly what is sent and to whom.
PROTO C
03 · C · DARK
Browser Field Kit
Browser-first visual QA — full-bleed remote page, summonable translucent tool trays, radial capture tool, hold-to-inspect DOM overlay, terminal as a bottom sheet.
PROTO D
04 · D · DARK
Timeline / Black Box
Event-sourced observatory — every session a replayable timeline of agent actions, human interventions, and failures; scrub back and branch a correction from history.
PROTO E
05 · E · BOLD
Deck of Agents
Tactile neubrutalist cards — one full-screen session at a time, horizontal deck, vertical layers, huge one-handed controls, flick a screenshot upward to attach it.
PROTO F
06 · F · OLED
Command Lens
Expert OLED minimalism — one universal command field over everything, structured shortcuts and natural language, staged multi-step voice actions awaiting confirmation.
PROTO G
07 · G · TABLET-FIRST
Remote Workbench
Calm industrial workspace — a genuine multi-pane tablet workbench with drag-and-drop evidence delivery, collapsing gracefully to a stacked phone layout with a mode rail.
Static prototypes only — no build step, no dependencies, fonts fall back to macOS-installed families. Every file renders the same fabricated fleet state and the same layered contexts (screen-home, screen-session, screen-terminal, screen-browser, screen-evidence, sheet-send) so directions can be compared and later ported.
